Overview:

To understand the system architecture a comprehensive suite of component diagrams are provided. One high-level component diagram which presents all the sub-systems are supported by four more detailed component diagrams which depict the J2EE patterns used and inter-component dependencies.

Use of Stereotypes:

In order to depict the design pattern and technology choices made stereotypes were heavily used in the component diagrams. No meta-model was created - pattern-based stereo types were used along with standard stereotypes for effective design communication:
  • <<StatelessSessionBean>> - stereotype representing the J2EE stateless session Enterprise JavaBean which communicates over RMI.
  • <<StatefulSessionBean>> - stereotype representing the J2EE stateful session Enterprise JavaBean which communicates over RMI.
  • <<LocalStatelessSessionBean>> - stereotype representing the J2EE local stateless session Enterprise JavaBean.
  • <<LocalEntityBean>> - stereotype representing the J2EE local entity Enterprise JavaBean which uses container managed persistence and relationships.
  • <<MessageDrivenBean>> - stereotype representing the J2EE message driven Enterprise JavaBean which listens to the JMS queue for messages.
  • <<ControllerServlet>> - stereotype representing the J2EE servlet implementation of the Front controller pattern.
  • <<Command>> - stereotype representing a Helper object designed based on the GoF command pattern.
  • <<JSP>> - stereotype representing the J2EE JavaServer Page.
  • <<UI>> - stereotype attached to a component representing a user interface.
  • <<DataTransferObject>> - stereotype representing a data transfer object (based on the pattern with the same name).
  • <<DataAccessObject>> - stereotype representing a data access object (based on the pattern with the same name).
  • <<ValueListHandler>> - stereotype representing a value list object (based on the pattern with the same name).
  • <<use>> - stereotype that signifies a usage dependency.
  • <<call>> - stereotype attached to a dependency within which the source invokes the target operation.
  • <<access>> - stereotype attached to a dependency to indicate that within the dependency the client package has the right to reference the contents of the supplier package.
  • <<send>> - stereotype attached to a usage dependency within which the source component sends a signal to the target component.
  • <<bind>> - stereotype attached to a dependency to signify a binding; where a binding is a dependency within which the source instantiates the target template to produce a new instance. Used primarily for services for document interchange.
  • <<queue>> - stereotype attached to a component which is a logical queue supporting asynchronous guranteed delivery of messages.
  • <<subsystem>> - stereotype attached to a package to signify that the package represents an independent part of the system being modelled.
  • <<system>> - stereotype attached to a package to signify that the package represents a system independent to the system being modelled.
  • <<tier>> - stereotype attached to a package to signfy that the package represents a logical tier.
